## 仓库管理命令

所有命令在 `ai_workspace/` 根目录运行。代理在 `tools/config.sh` 里配置（留空则不挂代理，该文件不进 git）。

### 配置代理

```bash
# 设置代理（同步更新 config.sh 和所有 prompts/ 里的示例地址）
./tools/set_proxy.sh http://127.0.0.1:20809

# 清除代理
./tools/set_proxy.sh
```

> `manage_repos.sh` 运行时自动从 `tools/config.sh` 读取代理，无需手动 export。

### 网络操作

**当前 shell 临时挂代理：**

```bash
export ALL_PROXY=http://127.0.0.1:20809
```

**git clone / submodule（单次）：**

```bash
ALL_PROXY=http://127.0.0.1:20809 git clone https://github.com/...
```

**HuggingFace 下载（二选一，不可共用）：**

```bash
# 方案 A：HF 镜像（无需代理）
HF_ENDPOINT=https://hf-mirror.com huggingface-cli download <org>/<model> \
    --local-dir /data1/flg/ckpt/<org>/<model>

# 方案 B：代理直连
ALL_PROXY=http://127.0.0.1:20809 huggingface-cli download <org>/<model> \
    --local-dir /data1/flg/ckpt/<org>/<model>
```

**清除代理（当前 shell）：**

```bash
unset ALL_PROXY https_proxy http_proxy
```

### 添加新 submodule

```bash
./manage_repos.sh add <路径|domain/name> <GitHub URL> [描述]
```

**路径简写**（research 仓库专用）：

```bash
# 简写 domain/name → research/domain/repos/name
./manage_repos.sh add embodied_ai/NewRepo \
    https://github.com/org/NewRepo.git "描述"

# 等价完整写法
./manage_repos.sh add research/embodied_ai/repos/NewRepo \
    https://github.com/org/NewRepo.git "描述"

# 嵌套子路径
./manage_repos.sh add generative_vision/dino/dinov4 \
    https://github.com/facebookresearch/dinov4.git "DINOv4"

# 添加自研项目（已有 GitHub 仓库时）
./manage_repos.sh add projects/myproject \
    https://github.com/Bismile/myproject.git "项目描述"
```

> 自动更新对应分区 README、research/README.md、根 README 目录表

### 移除 submodule

```bash
./manage_repos.sh rm <本地路径>
```

> 自动从各级 README 移除对应行

### 管理 Projects

Projects 有四种常见来源，统一放在 `projects/` 下管理：

---

**情形 1 — 全新自研：从零开始，推送到自己的仓库**

```bash
# 在本地创建项目并初始化 git
./manage_repos.sh new-project my-proj "项目描述"

# ... 本地开发 ...

# 在 GitHub 建好同名空仓库后，一键推送并注册为 submodule
./manage_repos.sh link-project my-proj \
    https://github.com/<YourOrg>/my-proj.git "项目描述"

# 后续提交推送
./manage_repos.sh push-project my-proj "feat: add new feature"
```

---

**情形 2 — clone 已有仓库：开发后推回原仓库（自己有写权限）**

```bash
# add 完成 clone + 注册（无需手动 git clone）
./manage_repos.sh add projects/some-repo \
    https://github.com/<Author>/some-repo.git "项目描述"

# 开发完成后推回原仓库
./manage_repos.sh push-project some-repo "fix: xxx"
```

---

**情形 3 — clone 已有仓库：改造后推送到自己的仓库（不走 fork）**

```bash
# 1. 先在 GitHub 建好自己的空仓库（如 <YourOrg>/my-somelib）
# 2. add 完成 clone + 注册（先挂原仓库地址）
./manage_repos.sh add projects/my-somelib \
    https://github.com/<Author>/some-repo.git "基于 some-repo 的自研版本"

# 3. 换成自己的 remote，保留原仓库为 upstream
cd projects/my-somelib
git remote set-url origin https://github.com/<YourOrg>/my-somelib.git
git remote add upstream https://github.com/<Author>/some-repo.git

# 4. 推送到自己的仓库
cd /data/flg/ai_workspace
./manage_repos.sh push-project my-somelib "initial commit"
```

---

**情形 4 — GitHub fork：在 fork 上开发，推送或贡献 PR**

```bash
# fork 的仓库直接用 add 注册（URL 填自己 fork 的地址）
./manage_repos.sh add projects/my-fork \
    https://github.com/<YourOrg>/my-fork.git "xxx（fork from <Author>/orig-repo）"

# 日常开发推送
./manage_repos.sh push-project my-fork "feat: update config"

# 同步上游最新代码
cd projects/my-fork
git remote add upstream https://github.com/<Author>/orig-repo.git  # 首次配置
git fetch upstream
git merge upstream/main
cd /data/flg/ai_workspace
./manage_repos.sh push-project my-fork "sync: merge upstream changes"

# 向原仓库贡献代码（PR 流程）
cd projects/my-fork
git checkout -b feat/my-feature
# ... 开发 ...
git push origin feat/my-feature
# 然后在 GitHub 上从 <YourOrg>/my-fork:feat/my-feature 向 <Author>/orig-repo:main 发起 PR

# 若有原仓库写权限，可直接推到原仓库分支
git push upstream feat/my-feature
```

---

> **已手动 `git clone` 的情况**：直接在已有目录上补注册：
> ```bash
> cd /data/flg/ai_workspace
> git submodule add --force <URL> projects/<项目名>
> ./manage_repos.sh readme && ./manage_repos.sh push "add: <项目名>"
> ```

```bash
python3 tools/sync_public.py
```

> 公开镜像：`Bismile/ai_workspace_public`，不含任何真实项目名或 URL

### 更新到最新 commit

**research 仓库**（拉取上游最新 commit）：

```bash
./manage_repos.sh update               # 全部（8 并行）
./manage_repos.sh update <本地路径>    # 单个
```

**clone 的项目**（原作者推了新代码，直接拉取）：

```bash
cd projects/some-repo
git pull                       # 等价于 git fetch + git merge origin/main
```

> 如果本地有修改而上游也改了同一处，会产生冲突，解决后 `git add . && git commit`。
> 若想放弃本地改动直接覆盖：`git fetch && git reset --hard origin/main`

**fork 项目**（把原仓库最新代码合并进来）：

```bash
# 首次：配置 upstream（只需做一次）
cd projects/my-fork
git remote add upstream https://github.com/<Author>/orig-repo.git

# 日常同步
cd projects/my-fork
git fetch upstream
git merge upstream/main        # 或 upstream/master，视原仓库而定
# 如有冲突，解决后 git add . && git commit

# 推送到自己的 fork 并更新主仓库指针
cd /data/flg/ai_workspace
./manage_repos.sh push-project my-fork "sync: merge upstream"
```

> 查看原仓库用的是哪个分支：`git remote show upstream`

### 重建所有 README 目录

```bash
./manage_repos.sh readme
```

从 `.gitmodules` 完整重建所有 README 的目录表（手动修改 submodule 后同步）。

### 提交并推送主仓库

```bash
./manage_repos.sh push                    # 默认消息
./manage_repos.sh push "feat: add xxx"    # 自定义消息
```

### 提交并推送某个 project 子仓库

```bash
./manage_repos.sh push-project <项目名> [commit消息]
```

> 自动 `git add -A && git commit && git push`，并同步主仓库的 submodule 指针（完整工作流见上方「管理 Projects」）

### 查看列表 / 状态

```bash
./manage_repos.sh list      # 列出全部 submodule 及 URL
./manage_repos.sh status    # 显示各 submodule 的 HEAD 状态
```
